What is ICMP?

ICMP stands for Internet Control Message Protocol. It is a network layer protocol. It communicates the network layer information between the end host and the router. It is a protocol, which basically helps us to troubleshoot the network issues that occur on the web server.


If any router in the way can’t send a request to the server because of some technical failure, then that router will send a message to the client, mentioning the reason for that failure. Unlike any other protocols, it is not for data transferring and receiving. It is designed for routers and other components that are connected to the Internet to talk about packet drops, routing and reachability issues.

But ICMP can’t correct any of the errors it detects, those functions are left for the higher layer protocols like TCP and UDP. However, with the help of ICMP messages, we can troubleshoot the problem it has encountered.

Construction of an IPv4 packets

Before getting into ICMP Service Model, we need to know the basics of packages. The request that you send to the server are in the form of packets. Size of each packet is 65535 bytes and the request that you make to the server or the data that you get back from the server are in the forms of several packets. Each packet has a header and the payload(in layman’s term, user data ). The header contains information like source destination, destination address, sequence number, protocols (UDP or TCP ), TTL, version(IPv4 or IPv6) etc. I will be explaining all these later in this article.

ICMP Server Model

ICMP service model is very simple. It basically sends the reporting error message from the router to the client, mentioning the reason for the error. But it is very unreliable, as it doesn’t have any checking mechanism to verify if the error packet has received to the sender.

Note: I am getting away by using phrases like “technical fault” or “ issue with the server ” while explaining why the user gets ICMP message. I might be doing that few more times. But in most cases it is not just server being offline, there are several other issues in the network. I will be going in depth about the other error messages in the 2nd article.

As discussed above, the packet has two parts, a header and the payload.

Header

I will not be discussing all of them, as some of them are not that relevant to this topic. Plus there are many websites, which have covered all of them. Hence it makes no sense covering them up.


Version: it carries the info whether it is IPv4 or IPv6.

Flags: If an IP Packet is too large to handle, these ‘flags’ tells if they can be fragmented or not. In this 3-bit flag, the MSB is always set to ‘0’.

Time to Live (TTL): To prevent looping of a packet in network, TTL is used. It has some set value which determines how many routers you can cross (hop). After every hop, its value is decremented by 1. So in case, if the package goes into a loop, its hop value will eventually decrease and would finally die.

Protocol: This tells if it is TCP or UDP.

Checksum : It is used to keep checksum value of entire header which is then used to check if the packet Source adress: This field contains the IP address of the user.

Destination address: This field contains the IP address of the server, we are sending request to.

Type: This tells the user, which type of error message it is. I have discussed the different types of error messages in the next section of this article.

Code: You can think of it as a subtype of the ‘Type’ discussed above. This takes a further step to describe the error, the user has encountered.

Payload Payload, also know as user data, consist of the IPv4 package header. This is the package, which encounters the ICMP error . Plus it has first 8bytes of the IPv4 package data.
